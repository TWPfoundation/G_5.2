G_5.2 milestone document
Release framing

I would define the release path like this:

Current state: operator-grade governed runtime
Next threshold: editorial-grade governed runtime
Next threshold: reflection-grade authored system
Release threshold: G_5.2 v1 / first complete usable system

This matches the repo’s current shape better than trying to jump straight from “working repo” to “public product.” The product brief is very clear that public launch is not the immediate goal .

Recommended execution order

This is the sequence I would actually recommend:

M0 — Baseline lock and source-of-truth cleanup
M1 — Persistence and trace hardening
M2 — Inquiry surface v1.5
M4 — Canon editorial workflow
M3 — Memory discipline v2
M5 — Reflection and authored artifact workflow
M6 — Eval and drift-control expansion
M7 — Operator studio integration
M8 — Release hardening and v1 threshold

The only non-obvious part is putting editorial workflow ahead of memory v2. My reason is simple: canon governance is the central missing organ. Memory can remain functional but partial for a while; canon evolution cannot. That ordering also stays aligned with the repo’s own self-conception that canon is primary, output is not canon unless promoted, and reflections become authoritative only through explicit human approval .

What I would call the actual v1 release gate

I would call G_5.2 v1 only when all of the following are true:

inquiry is reliable and inspectable
persistence is versioned and recoverable
memory has a governed lifecycle
canon proposals are first-class and reviewable
reflections can be authored without provenance contamination
evals cover runtime, memory, editorial flow, and provider drift
the operator can trace any important output back through its governing chain
the repo documentation matches reality
the system can be used regularly without manual patchwork
